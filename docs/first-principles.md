# Teacher 第一性需求

> 版本：2026-04-24  
> 目的：定义 Teacher 的产品原点。后续 memory、tools、voice、UI、pedagogy 的所有设计都必须服从这里的判断。

## 一句话定义

Teacher 不是一个“语音英语陪练 App”。Teacher 是一个 **会说话、会记忆、会写代码的语言学习 Agent**。

更精确地说：

> Teacher is a spoken interface to a persistent, coding-capable learning agent.

语音只是用户和 Agent 交互的自然入口；产品本体是一个能长期理解用户、主动规划学习、调用工具、生成教学软件、更新自己教学策略的 Agent。

## 第一性需求

### 1. Teacher 必须是 Agent，而不是被写死的程序

Teacher 的核心价值不在于“能和用户语音聊天”，而在于它能像一个真实助教一样持续行动：

1. 理解用户当前的学习目标和真实任务。
2. 读取长期学习记忆，知道用户过去的错误、词汇、偏好、薄弱点。
3. 判断这次对话最适合怎么练。
4. 在需要时调用工具或写代码，生成新的练习材料、测验、网页、复盘报告或教学界面。
5. 根据用户表现更新记忆和后续计划。

如果一个功能只是让 Teacher “说得更像人”，但没有增强它的记忆、行动、生成或自我调整能力，它不是核心需求。

### 2. 写代码是核心教学能力，不是附加功能

语言学习不是只能通过聊天完成。很多时候，最好的教学介质是临时生成的小工具：

- 一个 5 题的小测验页面
- 一个听力填空网页
- 一个面试 roleplay 控制台
- 一个词汇复习小游戏
- 一个 presentation timer + rubric
- 一个根据用户错误自动更新的 grammar trainer
- 一个展示用户错误趋势和复习计划的 dashboard

因此，Teacher 必须具备 coding-capable runtime：能创建文件、生成网页、运行代码、预览结果，并在用户反馈后继续修改。

这也是 Teacher 和普通语音学习产品的根本差异。普通产品把教学流程写死；Teacher 应该能为这一个用户、这一次学习目标，临时创造最合适的教学工具。

### 3. 记忆是产品内核，不是聊天增强

Teacher 的记忆不是“它记得你喜欢什么颜色”这种通用 personalization。Teacher 的记忆必须是教学状态：

- 用户的母语、目标语言、目标场景、真实任务
- 用户反复犯的错误，以及这些错误何时出现、是否改善
- 用户学过但还没掌握的词汇和表达
- 用户适合的纠错强度、语速、题型和教学风格
- 用户历史练习结果和下一步计划

每次交互都应该产生可写回的学习状态。没有记忆写回的对话，只是临时陪聊。

### 4. UI 应该由 Agent 按需生成，而不是完全预设

Teacher 可以有基础外壳，例如语音入口、当前任务面板、artifact 预览区、学习记忆区。但具体教学界面不应该全部预先写死。

当用户说“我明天要英文面试”，Teacher 应该能生成面试模拟器。  
当用户反复错介词，Teacher 应该能生成介词小测。  
当用户要练演讲，Teacher 应该能生成计时器、提纲卡片和评分 rubric。  
当用户说这个页面太难，Teacher 应该能修改题目、降级语言、增加提示。

产品的默认形态应该是：**稳定的 Agent 外壳 + Agent 动态生成的教学 artifacts**。

### 5. Voice-first 不等于 voice-only

Teacher 应该优先用语音和用户交流，因为语言学习的核心痛点是开口。但 voice-first 不意味着不能出现屏幕、网页、题目、图表或代码生成结果。

更准确的原则是：

- 用户的意图表达尽量语音化。
- 练习过程尽量让用户开口。
- 复杂反馈、题目、复盘和可视化可以显示在界面上。
- 键盘输入不是主要交互方式，但用户可以在需要时编辑文本、代码或题目。

## 整个项目的起点

项目的原始动机很简单：

1. 用户想学语言，尤其是想突破“能读能写，但开口不自然”的问题。
2. 现有语言学习产品大多是固定课程、固定练习、固定界面。
3. 通用聊天机器人虽然能对话，但缺少稳定的教学记忆和真实教学工作流。
4. 语音助手虽然能说话，但很多只是语音聊天壳，不会持续规划、不会自己创造练习工具。

进一步澄清后，真正的需求不是“做一个英语语音陪练”，而是：

> 如果 OpenClaw / Codex 这样的本地 Agent 能说话、能记住我的学习状态、能为了教我而写代码，它会比传统语言学习 App 更接近我真正想要的产品。

所以 Teacher 的起点是“把 coding agent 的创造能力带入语言学习”，而不是“给语言学习 App 加语音”。

## 调研结果

### 1. 市场不是空白，但核心缺口仍存在

调研显示，市场上已经有大量 AI 语言学习产品：

- Duolingo Max / Lily Video Call
- Speak
- ELSA
- Loora
- Praktika
- Langua
- Babbel Speak
- TalkPal / Talkio / Univerbal 等

这些产品正在快速补齐语音对话、个性化反馈、SRS、错题记录和学习报告。尤其 Langua 已经接近“教学记忆 + app 内 SRS”的方向。

因此，Teacher 不能建立在“别人都没有 AI 口语陪练”这个假设上。这个假设已经不成立。

真正仍然稀缺的是：

> 一个语言学习产品拥有真正 Agent runtime：长期记忆 + 工具调用 + 写代码 + 动态生成教学界面 + 持续调整学习策略。

### 2. 通用助手强，但不是语言学习 Agent

ChatGPT、Gemini、Claude 等通用助手已经能很好地解释语言、生成练习、进行 roleplay。它们是 Teacher 最大的替代品。

但它们的问题是：

- 学习记忆不是专门为语言学习设计的。
- 用户需要反复 prompt，才能得到稳定教学行为。
- 语音模式、工具模式、记忆模式、代码/artifact 生成模式经常不是同一个连续体验。
- 它们不会默认维护一个可审计的个人学习状态模型。

所以 Teacher 不能只比“英语学习 App”强；它必须比“用户自己用 ChatGPT/Gemini 学英语”更省心、更连续、更会行动。

### 3. 语音框架和记忆框架已经足够支撑 MVP

开源与开发者生态已经足够成熟：

- LiveKit Agents / Pipecat 可以支撑实时语音 agent。
- OpenAI Realtime / Agents SDK 可以支撑 voice + tools。
- Letta / mem0 / Zep / Graphiti 可以作为长期记忆基础设施。
- 本地代码执行、网页 artifact、浏览器预览可以组成 coding-capable teaching runtime。

技术瓶颈不是“能不能做语音对话”，而是产品架构是否从第一天就把 Teacher 当 Agent 设计。

## 深度思考

### 1. 语言学习的核心不是聊天，而是持续的个性化训练

聊天只是训练的一种形式。真正有效的学习需要：

- 重复暴露
- 主动输出
- 即时或延迟纠错
- 难度调节
- 场景迁移
- 复习计划
- 对错误的长期追踪

这些都要求系统有记忆、有判断、有行动，而不是只会响应当前一句话。

### 2. 教学材料应该被动态生成

传统语言学习 App 的问题是内容和界面被提前写死。用户的真实需求却高度动态：

- 明天要面试
- 下周要开会
- 今天听了一个 podcast
- 最近总说错某个表达
- 想练更自然的 small talk
- 想准备一次英文汇报

如果 Teacher 能写代码，它就不必等待产品团队提前开发所有练习形态。它可以在本次会话中生成本次需要的练习器。

这意味着 Teacher 的产品能力不是“拥有多少预设课程”，而是“能否为用户当下的学习问题生成合适的学习环境”。

### 3. Agent 的最小闭环

Teacher 的核心循环应该是：

1. 用户用语音表达需求或进行练习。
2. Agent 读取长期学习记忆。
3. Agent 判断当前最有价值的训练目标。
4. Agent 选择行动：继续对话、纠错、生成题目、写网页、调用工具、安排复习。
5. 用户完成练习。
6. Agent 评估表现。
7. Agent 更新记忆和下一步计划。

这个闭环比“语音输入 -> LLM 回复 -> TTS 输出”更重要。

### 4. Teacher 的护城河是学习状态和生成能力的结合

单独的语音不构成护城河。  
单独的记忆也不够，因为竞品会补。  
单独的代码生成也不够，因为通用 Agent 能做。

Teacher 的差异化在三者组合：

> 它知道你该练什么，并且能现场创造练习这个东西的工具。

这要求记忆模型和 code/artifact runtime 紧密连接。比如：

- `error_fossils` 触发 grammar quiz generator
- `vocab_srs` 触发 contextual speaking drill
- `calendar_events` 触发 meeting/interview simulator
- `session_results` 触发 dashboard 和下一周计划

### 5. 设计原则

后续所有需求都应该通过这几个问题筛选：

1. 这个功能是否让 Teacher 更像 Agent，而不是固定程序？
2. 它是否增强长期学习记忆？
3. 它是否让 Teacher 能生成或修改教学 artifact？
4. 它是否让用户更容易开口输出？
5. 它是否能写回可验证的学习进展？
6. 如果 ChatGPT/Gemini 加一句 prompt 就能做到，我们是否还提供了更连续、更低摩擦的体验？

## 非目标

Teacher 不应该被设计成：

- 又一个 ChatGPT 语音壳
- 又一个固定课程 App
- 又一个只做 Anki/SRS 的词汇工具
- 又一个只能 roleplay 的英语陪练
- 一个没有教学状态的通用 coding agent

Teacher 的目标不是覆盖所有语言学习功能，而是证明一个更强的范式：

> 语言学习产品可以不是固定软件，而是一个能为学习者持续生成软件的 Agent。

## MVP 含义

MVP 不应该只验证“语音能不能通”。这太低级。

MVP 应该验证：

1. Agent 能通过语音了解用户的真实学习任务。
2. Agent 能维护最小教学记忆：`user_profile`、`error_fossils`、`vocab_srs`。
3. Agent 能基于记忆生成一个可交互练习 artifact，例如小测验网页。
4. 用户完成 artifact 后，Agent 能评估结果并写回记忆。
5. 下一次语音会话中，Agent 能自然引用上次结果并调整练习。

如果这五件事成立，Teacher 就不是语音工具，而是学习 Agent。
