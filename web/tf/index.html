Vox 先做 Web App：Google 登录后，进入一屏自然对话课。

Vox 现在按多用户产品来做。每个用户先登录，再开始对话、练习和复盘；记忆、进度、用量都跟着自己的账号走。

Current Decision

Google 登录 + Supabase 记忆

第一版只做 Web App，不做 iOS 登录闭环。Google 负责确认身份，Vox 自己用早期白名单决定谁能进入，Supabase 保存每个用户自己的学习记录。

Web only iOS 暂停 先把网页端的登录、对话、练习、记忆跑顺，再评估 iOS。

登录方案 先解决能安全区分用户，再做更多登录方式。

1

Google 登录

用户用自己的 Google 账号进入 Vox。Vox 不保存密码，也不要求用户再记一套账号。

2

早期白名单

Google 登录不等于任何人都能用。第一阶段只允许白名单里的邮箱或域名进入。

3

用户记忆隔离

每个用户的偏好、对话总结、练习进度和用量都绑定到自己的账号，不和其他人混在一起。

为什么这样做 当前目标是最快拿到可靠的多用户 Web App。

摩擦最小 Google 登录比自建密码系统更快，也少了找回密码、密码安全、验证码滥用这些早期负担。
边界清楚 登录确认“你是谁”，白名单决定“你能不能进”。这两个问题分开，后面更容易扩展。
后续可加 如果真实用户被 Google 卡住，再加邮箱 magic link；现在先不为了假设需求增加复杂度。

上线前检查 避免登录能进、数据却串人的问题。

  1. 生产域名、预览域名、本地开发地址都要加入 Google 授权回调。
  2. 所有用户数据都必须带用户身份；没有身份的数据不能进入正式记忆。
  3. 白名单在后端判断，不能只靠前端隐藏按钮。
  4. 生产环境不保留免登录入口；测试入口只能用于本地或受控预览。
Vox Truthfile Web App first, iOS later.
